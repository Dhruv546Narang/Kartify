1. Vision & Problem Statement
1.1 The Problem
India's quick commerce market has exploded. As of 2024, platforms like Blinkit, Zepto, Swiggy Instamart, BigBasket Now, and JioMart all operate independently with their own apps, pricing, and checkout flows. A user buying groceries or essentials has to:
Open 3–5 different apps to compare prices for the same item.
Maintain separate carts on each platform.
Checkout multiple times with different payment flows.
Miss out on the best deal simply because they didn't check every app.
This is a fragmented, time-wasting experience. No unified layer exists for Indian quick commerce.
1.2 The Solution — Kartify
Kartify is a React Native mobile app that serves as a universal price comparison and cart aggregation layer on top of India's quick commerce platforms. Users search for a product once, see prices across all platforms, add items to a unified cart, and then intelligently checkout through the cheapest or fastest option.
Core value proposition in one line:
"Search once. Compare all. Checkout smart."

2.2 Why These Choices
React Native + Expo
Expo gives access to a managed workflow, OTA updates, and EAS Build for CI/CD — all without needing a Mac for iOS builds initially. React Native means one codebase for Android and iOS. Expo SDK 51+ is stable and battle-tested.
FastAPI
Async Python, automatic OpenAPI docs, Pydantic validation, blazing fast with uvicorn. Perfectly suited for an API that makes multiple external calls (to QuickCommerce API) and aggregates results quickly. Already familiar from prior projects.
Supabase
Managed Postgres with a REST API, real-time subscriptions, Row Level Security, and built-in auth. Replaces the need for a separate auth service and gives a dashboard for inspecting data. Free tier is generous.
Upstash Redis
Serverless Redis that works perfectly on Railway's ephemeral container model. Used for: caching QuickCommerce API responses (to avoid re-fetching for same product), rate limiting endpoints, and storing short-lived session data.
Railway
Zero-DevOps deployment for the FastAPI backend. Supports environment variables, automatic deploys from GitHub, and has a free starter plan. Much simpler than AWS or GCP for a solo developer.
3. Architecture
3.1 High-Level Architecture
The system has three main layers:
Layer
Components
Client (Mobile)
React Native app — handles UI, local state, auth tokens, and user interactions.
Backend (API)
FastAPI server — orchestrates QuickCommerce API calls, handles auth, applies business logic, writes/reads Supabase, uses Redis cache.
Data Layer
Supabase (persistent user data, carts, history) + Upstash Redis (ephemeral cache + rate limits) + QuickCommerce API (real-time price data).
3.2 Request Flow — Product Search
A typical search request flows as follows:
User types a product name in the search bar on the React Native app.
App sends GET /search?q=<query> to the FastAPI backend with auth token.
FastAPI checks Upstash Redis cache for this query. If found (cache hit), return immediately.
If cache miss: FastAPI calls QuickCommerce API with the query.
QuickCommerce API returns prices from Blinkit, Zepto, Instamart, BigBasket, JioMart.
FastAPI normalizes and sorts the response, writes to Redis cache (TTL: 10 minutes).
FastAPI returns the unified price list to the React Native app.
App displays results as a ranked list with platform badges and price highlights.

4.4 Caching Strategy
Redis (Upstash) is used for two purposes:
Search Cache: Key = search:<query_normalized>, TTL = 10 minutes. Avoids repeated QuickCommerce API calls for the same product.
Rate Limiting: Key = ratelimit:<user_id>, with a sliding window counter. Prevents API abuse. Limit: 60 requests/minute per user.
Cache invalidation is time-based only (TTL expiry). No manual invalidation needed since price data changes naturally within minutes.

5.2 Screen-by-Screen Breakdown
HomeScreen
Landing screen after login. Shows: a large search bar at the top, recent searches from search history, quick-access to the current active cart, and a summary of top platform deals of the day (can be mocked initially).
SearchScreen
The core screen. User types a query. Results are fetched via GET /search. Results rendered as a list of ProductCards showing the product name and a price comparison row across platforms. Tapping a result opens ProductDetailScreen.
ProductDetailScreen
Shows one product with a full PriceTable: all platform prices ranked cheapest to most expensive, delivery ETAs (if available from API), and an Add to Cart button that lets the user pick quantity and platform preference. Can also compare price trend if history is stored.
CartScreen
Shows all items in the current cart, grouped by platform. Displays total per platform, and a smart suggestion for the cheapest combination checkout strategy. Button to initiate checkout.
CheckoutScreen
Not a real in-app checkout (deep linking to the platform's app is the MVP approach). Shows the checkout breakdown: which items from which platform, totals, and deep link buttons to open Blinkit/Zepto/etc. with the items prefilled (where API allows).
HistoryScreen
List of past searches. Tapping re-runs the search. Can be cleared.
LoginScreen / SignupScreen
Standard email/password forms. Calls /auth/login and /auth/signup. Stores JWT in Expo SecureStore. Handles error states gracefully.
5.3 Navigation Structure
Two navigators are used:
Auth Stack: Login → Signup (shown when user is not authenticated)
App Tab Navigator (shown when authenticated): Home | Search | Cart | History
Stack navigator nested inside App tabs for: Search → ProductDetail, Cart → Checkout
5.4 State Management with Zustand
Two Zustand stores:
authStore: holds { user, token, isLoggedIn }, actions: login(), logout(), loadFromStorage()
cartStore: holds { carts, activeCartId }, actions: addItem(), removeItem(), setActiveCart()
cartStore syncs with the backend on every mutation. Optimistic updates are used for a snappy UI, with rollback on error.
6. QuickCommerce API Integration
6.1 What It Provides
The QuickCommerce API is a third-party aggregator that provides product search results and pricing data across Indian quick commerce platforms (Blinkit, Zepto, Swiggy Instamart, BigBasket Now, JioMart). It abstracts away the need to scrape or individually integrate with each platform.
6.2 Backend Integration Pattern

async def search_products(query: str) -> list[dict]:
    async with httpx.AsyncClient() as client:
        response = await client.get(
            f"{QUICKCOMMERCE_BASE_URL}/search",
            headers={"Authorization": f"Bearer {QUICKCOMMERCE_API_KEY}"},
            params={"q": query},

6.4 Verification Steps (Before Starting Dev)
Sign up for QuickCommerce API and obtain API key.
Verify the base URL and authentication method (Bearer token).
Run a test search request from terminal using curl to confirm response shape.
Map actual API response fields to the unified schema above.
Set QUICKCOMMERCE_BASE_URL and QUICKCOMMERCE_API_KEY in .env.
Update normalize_results() function to match actual response structure.

8. Known Risks & Mitigation
Risk
Mitigation
QuickCommerce API rate limits or downtime
Redis caching minimizes API calls. Add fallback mock data for demo purposes.
QuickCommerce API schema changes
Keep normalize_results() isolated. Unit test it separately.
Deep links not working on all platforms
Test each platform. Fall back to opening the platform's web URL if deep link fails.
Expo version mismatches (seen before)
Pin SDK version in app.json. Don't upgrade mid-development.
Railway cold starts
Use Railway's always-on plan if needed, or add a keep-alive ping.
Supabase RLS misconfiguration
Test RLS policies thoroughly. Use Supabase dashboard to verify data isolation.
9. Future Features (Post-MVP)
9.1 Smart Cart Optimizer
Given a cart of N items, find the optimal split across platforms that minimizes total cost + delivery fee. This is a combinatorial optimization problem — can use a greedy algorithm or even a simple LP solver.
9.2 Price Drop Alerts
Users can watch a product and get a push notification (via Expo Notifications + a backend cron job) when its price drops below a threshold on any platform.
9.3 Shared Carts
Multiple users (e.g. a household) can share a cart. Built on Supabase real-time subscriptions.
9.4 AI Deal Recommender
An AI layer (Claude API) that looks at a user's purchase history and proactively recommends deals or restocking reminders. ('You usually buy milk every 5 days — Zepto has it 20% off today.')
9.5 Platform Wallet/Coupon Tracking
Let users input their wallet balances on each platform, so the checkout optimizer can factor in effective prices after coupon/cashback applied.
9.6 Monetization Path
Affiliate commissions from platforms when users checkout through Kartify.
Premium subscription: price history graphs, unlimited price alerts, family cart sharing.
B2B: White-label the comparison engine for grocery deal aggregator sites.
